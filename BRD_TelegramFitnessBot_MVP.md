# BRD (Черновик) — Telegram Fitness Bot (MVP)

**Заказчик:** Господин  
**Канал:** Telegram Bot (кнопки + минимальный числовой ввод)  
**Режим пользователей:** мультипользовательский, но **каждый сам по себе** (данные изолированы по Telegram `user_id`)  
**Цели MVP:**
- Удобно пользоваться **прямо во время тренировки**.
- Вести учёт: тренировки (упражнения/подходы/вес/повторы + отдых), прогресс тела (вес/обхваты/фото), питание (простое меню на день по КБЖУ).
- Получать отчёт **по кнопке** за выбранный период.

---

## 1) In-scope MVP (Must Have)

### 1.1 Онбординг + профиль пользователя
**Описание:** при первом запуске бот проводит “знакомство” (опрос) и сохраняет базовый профиль.

**Поля профиля (минимум):**
- Telegram `user_id`, `chat_id`
- пол (опц.), возраст (опц.), рост, вес (текущий)
- цель: жиросжигание / поддержание / набор
- частота тренировок: 3/4/5+ раз в неделю
- уровень: новичок/средний/продвинутый
- ограничения/травмы (свободное поле, опц.)
- доступное оборудование: зал/дом (опц.)

**Критерии приёмки:**
- Онбординг запускается командой `/start` или кнопкой «Начать».
- После завершения анкеты пользователь видит главное меню.

### 1.2 Генерация плана тренировки (GPT) по анкете + каталог упражнений
**Описание:** после онбординга бот генерирует **персональный план тренировок** по дням. Важно: LLM используется **только на старте** (генерация плана/подбор упражнений), а дальше система переиспользует упражнения из БД.

**Требование к данным (обязательно):**
- Отдельная сущность **Exercise** (упражнение) с понятной структурой (название, группа мышц/категория, оборудование, варианты выполнения и т.п.).
- Отдельная сущность **TrainingPlan** (план).
- План **составляется из существующих упражнений** (references → Exercise), а не из “текста”.
- При генерации плана новые упражнения (если их нет) **сохраняются в БД упражнений** для дальнейшего переиспользования и нормализации.

**MVP-ограничение:**
- План генерируется и сохраняется как “активный” (без сложной периодизации).
- Пользователь может **корректировать** план (см. 1.3) без повторного прохождения анкеты.

**Критерии приёмки:**
- После анкеты бот выдаёт план на неделю/цикл (минимум: 4 дня или выбранное число дней).
- План хранится и доступен в «Планы тренировок».
- В БД есть таблица/коллекция упражнений; элементы плана ссылаются на упражнения по ID.
- Повторная генерация/корректировка не требует постоянного участия LLM: упражнения выбираются из каталога, LLM нужен только для первичного подбора/расширения каталога.

### 1.3 Тренировки во время занятия: выбор упражнения, подходы, отдых
**Описание:** ключевой сценарий — вести тренировку в реальном времени.

**Функции:**
- «Начать тренировку» → выбор дня/тренировки из активного плана.
- Экран упражнения:
  - выбор упражнения из плана (кнопками)
  - «Ввести подход» → ввод *вес* и *повторы*
  - авто-нумерация подходов
  - кнопка‑toggle «Начать отдых / Завершить отдых» (без ручного таймера)
  - «Следующее упражнение»
  - «Завершить тренировку»
- Редактирование логов тренировки (минимум: в день внесения):
  - исправить вес/повторы
  - удалить/добавить подход

**Критерии приёмки:**
- Тренировку можно провести без свободного текста: кнопки + числовые поля.
- Данные сохраняются раздельно по пользователям.
- Можно прервать тренировку и продолжить позже.

### 1.4 Прогресс тела: вес, обхваты, фото
**Функции:**
- «Новый замер»:
  - вес
  - обхваты (фиксированный набор полей, часть можно пропускать)
- «Фото прогресса»:
  - пользователь отправляет фото, бот привязывает к дате
- «История»:
  - за 7/14/30 дней (кнопки)

**Критерии приёмки:**
- Замеры вводятся быстро (≤5 кликов + 1–2 поля).
- Фото сохраняется и доступно из истории.

### 1.5 Питание: генерация меню на 1 день
**Описание:** простая генерация “меню на день” под целевые КБЖУ.

**Функции:**
- «Профиль питания»:
  - целевые калории и/или БЖУ (ввод пользователем; автоподсказки допустимы)
  - количество приёмов пищи в день (3–6)
- «Меню на день»:
  - генерируется 1-дневное меню
  - кнопка «Сгенерировать ещё»
  - (опц.) «Сохранить»

**MVP-упрощения:**
- Без продвинутой базы продуктов и точных рецептов.
- Допускается приблизительность (например, попадание по калориям ±10–15%).

### 1.6 Питание: учёт потреблённых калорий (лог)
**Описание:** помимо генерации меню, в MVP нужен **учёт фактически потреблённых калорий**.

**Функции (MVP):**
- Пользователь добавляет записи вида: дата + калории (+ опционально комментарий).
- Отображение списка записей за период (7/14/30 дней).

**В будущем (out-of-scope MVP):**
- Определение калорий/состава еды **по фото**.

### 1.7 Отчёт (по кнопке) за период
**Описание:** отчёт формируется по запросу, период выбирается кнопками.

**Периоды (фиксированные):** 7 / 14 / 30 дней.

**Состав отчёта (минимум):**
- тренировки: кол-во тренировок за период; краткая статистика (подходы/объём в упрощённом виде)
- вес: изменение (первое/последнее значение за период)
- замеры/фото: наличие, количество, последние даты
- питание: (а) сколько дней было меню (если генерировалось), (б) учёт потреблённых калорий — среднее/сумма за период (в MVP упрощённо)

## 2) Out-of-scope для MVP (v1+)
- Тренерский режим (доступ к данным других пользователей)
- Авто-рассылка отчётов по расписанию
- Сложная периодизация/циклы 4–12 недель
- Графики (можно позже)
- Интеграции (Google Sheets/Notion/браслеты)
- Полноценная база продуктов/рецептов с точными граммовками
- Определение калорий/состава еды по фото

## 3) UX: главное меню и основные экраны (кнопки)
**Главное меню:**
- Тренировка
- Прогресс
- Питание
- Отчёт
- Настройки

**Тренировка:**
- Начать тренировку
- Продолжить тренировку
- Планы тренировок

**Прогресс:**
- Новый замер
- Фото прогресса
- История (7/14/30)

**Питание:**
- Профиль питания
- Меню на день
- Сгенерировать ещё
- Калории (добавить запись)
- Калории (история)

**Отчёт:**
- 7 дней
- 14 дней
- 30 дней

---

## 4) Нефункциональные бизнес‑требования
- Приватность: данные одного пользователя не видны другим.
- Удаление данных пользователя (желательно v1, но закладываем в модель).
- Надёжность: бот должен корректно работать для десятков пользователей.
- Простота ввода: минимум текста, максимум кнопок.
